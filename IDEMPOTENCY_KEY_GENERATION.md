The server treats the header as an opaque string — it doesn't validate format, length, or charset. So the BFF gets to define the contract.

  Recommended approach: UUID v4 per logical intent, generated by the client

  The key tracks a user intent, not a network call. Walk through what happens when things go wrong, and the right placement falls out:

  ┌────────────────────────────────────┬─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┐
  │     Where the key is generated     │                                                            What happens on retry                                                            │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ Frontend (web/mobile) at form open │ Frontend keeps the same UUID across retries. BFF passes it through. ✅ Correct dedup.                                                       │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ BFF on each incoming request       │ BFF generates a fresh UUID each call. Frontend retry → new UUID → API treats as a new product. ❌ Defeats the point.                        │
  ├────────────────────────────────────┼─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┤
  │ BFF with session-scoped dedup      │ BFF must persist (user, form_id) → UUID across retries. Doable but you're rebuilding the frontend's job in the BFF. ⚠️ Works but redundant. │
  └────────────────────────────────────┴─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────┘

  Generate on the frontend; pass through the BFF unchanged.

  What the frontend does

  // Web (browsers ≥ Chromium 92 / Safari 15.4 / Firefox 95)
  const idempotencyKey = crypto.randomUUID();

  // React Native / older browsers
  import { v4 as uuidv4 } from 'uuid';
  const idempotencyKey = uuidv4();

  Persist it for the lifetime of the form, not the lifetime of the session:

  const [idempotencyKey] = useState(() => crypto.randomUUID());

  const submit = async () => {
    const res = await fetch('/api/products', {
      method: 'POST',
      headers: { 'X-Idempotency-Key': idempotencyKey, ... },
      body: JSON.stringify(formData),
    });
    // On 5xx: retry the fetch with the SAME idempotencyKey
    // On 2xx or 4xx: discard the key (or remount the form to generate a new one)
  };

  Rule of thumb: the key lives as long as the user is trying to commit the same intent. As soon as the API gives a definitive answer (201 or 4xx), the key is spent.

  What the BFF does

  Just forward. Two concerns worth handling:

  1. Always forward when present, never invent silently.

  // BFF route handler
  app.post('/api/products', async (req, res) => {
    const idempotencyKey = req.header('x-idempotency-key');
    if (!idempotencyKey) {
      // Reject — refuse to silently call a write API without an idem key.
      return res.status(400).json({ error: 'X-Idempotency-Key required' });
    }

    const upstream = await fetch(`${API}/v1/products`, {
      method: 'POST',
      headers: {
        'Idempotency-Key': idempotencyKey,
        // ... auth, org context, content-type
      },
      body: req.rawBody, // pass the original bytes — see below
    });
    // Mirror status + body
  });

  2. Forward the request body byte-for-byte.

  The server hashes sha256(body) and compares against the cached entry on retries. If your BFF deserializes JSON and re-serializes it, the hash changes (field order, whitespace). A
  legitimate retry would then trip 422.

  In Node/Express, capture req.rawBody via a body-parser hook. In Go, hand the upstream the same []byte you read. Don't transform between BFF and API for an idem-keyed request.

  If the BFF must transform (adds fields, rewrites IDs), the BFF needs to:
  - Run its own dedup layer keyed by the frontend's X-Idempotency-Key, OR
  - Generate its own stable BFF-side Idempotency-Key (e.g., sha256(frontendKey + bffTransformVersion)).

  For a clean pass-through BFF, neither is needed.

  What to generate if the client truly can't

  If you're calling the API from a script, cron job, or system that has no "form open" event, generate one key per logical operation and persist it before sending:

  // Go example: cron-style ingestion script
  key := uuid.NewString()
  if err := persistIdempotencyKeyForJob(jobID, key); err != nil { ... }
  // Use `key` for every retry of this specific job, even across process restarts.

  The key has to outlive a process crash; otherwise restart-induced retries create duplicates.

  Gotchas

  - Don't hash the payload as the key. Sounds tempting ("deterministic, free, no persistence needed") but means two legitimately-different create requests with byte-identical bodies
  collide — and the API returns the first one's response. Use a UUID; the payload hash is the server's job.
  - Don't reuse keys across products. New form → new key.
  - 24h horizon. The Redis cache is 24h; the DB row holds the key forever. So a retry within 24h gets the fast-path 200; a retry after 24h with a cleared Redis still hits the DB and gets
  200 (your controlled-clients deal makes this safe). But the frontend should never voluntarily retry a 24-hour-old key — that suggests the user re-clicked "Submit" from a stale tab.
  - Log the key on both sides. When debugging "duplicate product" reports, the idempotency key is the join column between frontend, BFF, and API logs.

  Two small server-side improvements worth doing eventually

  Not asking, just flagging since this is in the area:

  1. Length cap. The server currently accepts arbitrary-length keys. A 1MB header is denial-of-service territory. Reject keys > 255 chars (matches Stripe's limit). One-line check in
  CreateProduct.
  2. Format validation. Reject empty-but-present keys (Idempotency-Key: ), keys with control characters, etc. Tightens the contract.
